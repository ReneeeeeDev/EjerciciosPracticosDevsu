CONCLUSIONES Y HALLAZGOS DEL EJERCICIO DE AUTOMATIZACIÓN

================================================================================
RESUMEN EJECUTIVO
================================================================================

Este documento presenta mis hallazgos y conclusiones del desarrollo de los 
ejercicios de automatización E2E y API Testing para el proceso de selección.

Ejercicios completados:
- Automatización E2E: SauceDemo (Opción 2)
- Automatización API: Petstore (Opción 2)

Framework utilizado: Cypress 13.6.0
Fecha de desarrollo: Febrero 2026

================================================================================
DECISIONES TÉCNICAS
================================================================================

Selección de opciones
----------------------

E2E - Opción 2 (SauceDemo):
Elegí SauceDemo porque es una aplicación diseñada específicamente para 
testing automation. Durante mi investigación previa, noté que tiene selectores 
muy bien definidos con atributos data-test, lo cual facilita el mantenimiento 
de las pruebas a largo plazo. Además, la aplicación es muy estable y no 
presenta problemas de timeouts o inconsistencias.

API - Opción 2 (Petstore):
Seleccioné Petstore principalmente por su documentación completa en Swagger. 
La API está bien diseñada y permite probar un flujo CRUD completo, lo cual 
demuestra mejor mis habilidades en testing de APIs. También me gustó que 
tiene endpoints para búsqueda filtrada, lo que permite hacer validaciones 
más complejas.

Framework - Cypress:
Decidí usar Cypress por varias razones:
1. Es uno de los frameworks mencionados como deseables en las instrucciones
2. Me permite hacer tanto E2E como API testing en un solo framework
3. Tiene excelente debugging con la funcionalidad de time-travel
4. Genera screenshots y videos automáticamente, lo cual es valioso para 
   documentar fallos
5. La configuración inicial es mínima comparado con Selenium

Consideré usar Selenium + Karate (uno para E2E y otro para API), pero 
preferí mantener todo en un solo framework para simplificar la estructura 
del proyecto y el mantenimiento futuro.

================================================================================
PRUEBAS E2E - SAUCEDEMO
================================================================================

Implementación
--------------

Desarrollé 3 escenarios de prueba que cubren:

1. Flujo completo de compra (happy path)
   - Autenticación con usuario válido
   - Selección de 2 productos (Backpack y Bike Light)
   - Verificación del carrito
   - Proceso de checkout completo
   - Validación del mensaje de confirmación

2. Validación de carrito vacío
   - Verificar que el checkout no está disponible sin productos

3. Validación de login fallido
   - Probar con credenciales incorrectas

Hallazgos durante el desarrollo
--------------------------------

Aspectos positivos:
- Los selectores data-test son muy confiables. Durante todas las ejecuciones
  no tuve ni un solo problema de selectores que cambien o fallen.
  
- El sitio es muy rápido. Los tiempos de carga son consistentes entre 1-2 
  segundos, lo cual permite que las pruebas se ejecuten rápidamente.

- La lógica del carrito es sólida. Los contadores se actualizan correctamente
  y los productos se mantienen en el carrito durante toda la sesión.

Observaciones:
- No hay validación real de tarjeta de crédito (es simulado), lo cual es 
  esperado para un sitio de práctica.
  
- Todos los usuarios tienen acceso al mismo inventario. En un ambiente real,
  esto podría ser diferente según roles o ubicaciones.

- La aplicación no tiene manejo de sesiones complejas. Una vez que se hace
  login, no hay expiración de sesión ni refresh de token.

Posibles mejoras:
Si tuviera más tiempo, agregaría:
- Validación de los cálculos de precio (subtotal + tax = total)
- Pruebas con los diferentes tipos de usuario (problem_user, locked_out_user)
- Validación del sorting de productos
- Pruebas de remover productos del carrito

Métricas
--------
- Tests implementados: 3
- Tiempo de ejecución total: ~30 segundos
- Tasa de éxito: 100%
- Líneas de código: ~150

================================================================================
PRUEBAS API - PETSTORE
================================================================================

Implementación
--------------

Desarrollé 7 escenarios de prueba cubriendo:

1. POST /pet - Crear una nueva mascota
2. GET /pet/{id} - Consultar mascota por ID
3. PUT /pet - Actualizar nombre y status
4. GET /pet/findByStatus - Buscar por status "sold"
5. Flujo CRUD completo (Create, Read, Update, Delete)
6. Manejo de errores (404 Not Found)
7. Validación de schema de respuesta

Hallazgos durante el desarrollo
--------------------------------

Aspectos positivos:
- La API es muy consistente en sus respuestas. Todos los status codes son
  apropiados (200 para success, 404 para not found, etc.)

- La documentación en Swagger es excelente. Pude entender rápidamente la
  estructura de los requests y responses.

- Los tiempos de respuesta son buenos (<500ms en promedio), lo cual permite
  ejecutar muchas pruebas rápidamente.

Observaciones interesantes:
- Puedes enviar tu propio ID en el POST. Inicialmente pensé que el servidor
  siempre generaría uno automático, pero respeta el ID que envías si lo 
  incluyes en el body. Usé timestamps para generar IDs únicos.

- Los datos no persisten indefinidamente. Noté que mascotas creadas en 
  ejecuciones anteriores a veces desaparecen. Por esto decidí que cada test
  cree sus propios datos en lugar de depender de datos existentes.

- El endpoint findByStatus puede retornar arrays muy grandes. Cuando busqué
  mascotas con status "available", obtuve cientos de resultados. Por eso mi
  validación solo verifica que MI mascota esté en el array.

Decisiones de diseño:
- Cada test crea sus propios datos. Esto hace que los tests sean 
  independientes y puedan ejecutarse en cualquier orden.

- Usé cy.request() en lugar de crear una capa de abstracción más compleja.
  Para un proyecto de este tamaño, cy.request() es suficiente y mantiene
  el código simple.

- Implementé validaciones de schema además de validaciones de valores 
  específicos. Esto ayuda a detectar cambios breaking en la API.

Métricas
--------
- Tests implementados: 7
- Endpoints probados: 4 diferentes
- Métodos HTTP: GET, POST, PUT, DELETE
- Tiempo de ejecución total: ~10 segundos
- Tasa de éxito: 100%
- Líneas de código: ~200

================================================================================
ARQUITECTURA Y ORGANIZACIÓN
================================================================================

Estructura del proyecto
-----------------------

Decidí organizar el proyecto de la siguiente manera:

cypress/
  e2e/        - Pruebas End-to-End
  api/        - Pruebas de API
  support/    - Comandos personalizados y configuración
  fixtures/   - Datos de prueba

Esta separación facilita:
- Ejecutar solo E2E o solo API según necesidad
- Mantener el código organizado por tipo de prueba
- Escalar el proyecto agregando más pruebas en el futuro

Comandos personalizados
------------------------

Creé algunos comandos reutilizables en support/commands.js:
- loginSauceDemo(): Para evitar repetir el login en múltiples tests
- createPet(), updatePet(), etc.: Para operaciones comunes de API

Aunque en este proyecto pequeño no se usan extensivamente, en un proyecto
real estos comandos serían muy útiles para evitar duplicación de código.

================================================================================
MEJORES PRÁCTICAS APLICADAS
================================================================================

Durante el desarrollo, intenté seguir varias mejores prácticas:

1. Tests independientes
   - Cada test puede ejecutarse por separado
   - No dependen del orden de ejecución
   - Cada test crea sus propios datos de prueba

2. Selectores robustos
   - Preferí data-test attributes sobre selectores CSS frágiles
   - Evité usar selectores basados en posición (nth-child, etc.)
   - Usé clases semánticas cuando los data-test no estaban disponibles

3. Validaciones completas
   - No solo verifico que algo existe, sino que tiene el valor correcto
   - Valido status codes en todas las llamadas API
   - Verifico la estructura de las respuestas (schemas)

4. Código legible
   - Comentarios descriptivos en secciones complejas
   - Nombres de variables y funciones claros
   - Mensajes cy.log() para facilitar debugging

5. Manejo de datos dinámicos
   - Uso timestamps para generar IDs únicos
   - No hardcodeo valores que pueden cambiar
   - Variables compartidas para datos reutilizados

================================================================================
DESAFÍOS Y APRENDIZAJES
================================================================================

Desafíos encontrados
--------------------

1. Datos no persistentes en Petstore
   Problema: Al principio intenté reutilizar mascotas entre tests, pero
   descubrí que los datos se limpian periódicamente.
   Solución: Cada test crea sus propios datos.

2. Arrays grandes en findByStatus
   Problema: El endpoint puede retornar cientos de resultados.
   Solución: Usar .find() para buscar específicamente mi mascota en el array.

3. Tiempos de espera
   Problema: Algunos elementos en SauceDemo tardan en cargar.
   Solución: Cypress maneja esto automáticamente con su sistema de retry,
   pero aumenté el timeout en cypress.config.js como precaución.

Aprendizajes
------------

1. Cypress maneja automáticamente muchas cosas que en Selenium requieren
   código explícito (waits, promises, retry logic).

2. Para API testing, cy.request() es suficiente para la mayoría de casos.
   No necesitas una librería adicional como Axios.

3. La documentación es clave. Swagger me ahorró mucho tiempo en entender
   la estructura de la API de Petstore.

4. Tests independientes son más valiosos que tests rápidos. Aunque crear
   datos en cada test toma más tiempo, la confiabilidad vale la pena.
